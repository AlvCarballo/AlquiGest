Migrar a un servidor web

Todo lo que debes cambiar en el código antes de publicar AlquiGest en internet o en una red corporativa.

CORS Autenticación HTTPS PHP Seguridad

¿Qué cambia al pasar a un servidor web?

Actualmente AlquiGest está diseñado para funcionar solo en localhost. El código tiene varias capas de protección que asumen que nadie más puede acceder a la máquina. Al moverlo a un servidor público esas suposiciones dejan de ser válidas.

⚠ En localhost (actual)
  • Solo tú tienes acceso a la máquina
  • No hay contraseñas de usuario en la app
  • La API solo responde a 127.0.0.1
  • HTTP sin cifrar es aceptable
  • install.php solo lo ves tú
  • Credenciales root/root en la BD
✓ En servidor web (objetivo)
  • Cualquiera puede intentar acceder
  • Login obligatorio para usar la app
  • CORS restringido a tu dominio
  • HTTPS obligatorio (certificado SSL)
  • install.php eliminado o bloqueado
  • Usuario de BD con permisos mínimos

Qué es CORS y por qué importa

CORS (Cross-Origin Resource Sharing) es un mecanismo del navegador que controla qué dominios pueden llamar a tu API desde JavaScript.

Cuando AlquiGest está en https://tuapp.com y su JavaScript llama a https://tuapp.com/api.php, es una petición del mismo origen y no necesita CORS. El problema surge cuando el cabecero Access-Control-Allow-Origin es demasiado permisivo.

Paso 1 — Activar HTTPS Crítico

🔒
Sin HTTPS todo lo demás da igual.

Sin cifrado, las credenciales de acceso, los datos de inquilinos y los IBANs viajan en texto plano por la red. Cualquiera en la misma red (WiFi de oficina, router corporativo) puede interceptarlos.

Cómo obtener HTTPS gratuito con Let's Encrypt

Si tu servidor es Linux con Apache o Nginx, Certbot instala un certificado SSL válido en menos de 5 minutos:

Terminal del servidor Shell 
# Ubuntu / Debian con Apache
sudo apt install certbot python3-certbot-apache
sudo certbot --apache -d tudominio.com -d www.tudominio.com

# Ubuntu / Debian con Nginx
sudo apt install certbot python3-certbot-nginx
sudo certbot --nginx -d tudominio.com -d www.tudominio.com

Forzar HTTPS con .htaccess (Apache)

Crea o edita el archivo .htaccess en la raíz del proyecto para redirigir todo el tráfico HTTP a HTTPS:

.htaccess Nuevo / Editar 
RewriteEngine On

# Redirigir HTTP → HTTPS
RewriteCond %{HTTPS} off
RewriteRule ^ https://%{HTTP_HOST}%{REQUEST_URI} [L,R=301]

# Bloquear acceso directo a install.php desde navegador
RewriteRule ^install\.php$ - [F,L]

# Cabeceras de seguridad HTTP
Header always set Strict-Transport-Security "max-age=31536000"
Header always set X-Content-Type-Options "nosniff"
Header always set X-Frame-Options "DENY"

Paso 2 — Añadir autenticación Crítico

🚨
La app actualmente no tiene ningún sistema de login.

En un servidor público, cualquier persona que conozca la URL puede ver y modificar todos los datos. Debes añadir autenticación antes de cualquier otra cosa.

1. Crear la tabla de usuarios en MySQL

install.php — añadir a los CREATE TABLE Añadir 
CREATE TABLE `usuarios` (
    `id`         INT AUTO_INCREMENT PRIMARY KEY,
    `nombre`     VARCHAR(100) NOT NULL,
    `email`      VARCHAR(150) NOT NULL UNIQUE,
    `password`   VARCHAR(255) NOT NULL,  -- hash bcrypt
    `rol`        ENUM('admin','lectura') DEFAULT 'admin',
    `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

2. Crear login.php

login.php Nuevo archivo 
<?php
session_start();
header('Content-Type: application/json; charset=utf-8');

if (isset($_SESSION['user_id'])) {
    echo json_encode(['ok' => true]);
    exit;
}

if ($_SERVER['REQUEST_METHOD'] === 'POST') {
    $cfg   = require __DIR__ . '/assets/php/config.php';
    $input = json_decode(file_get_contents('php://input'), true) ?: [];
    $email = trim($input['email'] ?? '');
    $pass  = $input['password'] ?? '';

    $pdo = new PDO(
        "mysql:host={$cfg['host']};dbname={$cfg['name']};charset=utf8mb4",
        $cfg['user'], $cfg['pass'],
        [PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION]
    );
    $st = $pdo->prepare('SELECT * FROM usuarios WHERE email = ? LIMIT 1');
    $st->execute([$email]);
    $user = $st->fetch(PDO::FETCH_ASSOC);

    if ($user && password_verify($pass, $user['password'])) {
        session_regenerate_id(true); // previene session fixation
        $_SESSION['user_id']  = $user['id'];
        $_SESSION['user_rol'] = $user['rol'];
        echo json_encode(['ok' => true]);
    } else {
        http_response_code(401);
        echo json_encode(['error' => 'Credenciales incorrectas']);
    }
    exit;
}

3. Proteger api.php con sesión

Sustituye el bloque de restricción de IP por la comprobación de sesión. El bloque actual dice "solo localhost"; en producción debe decir "solo usuarios autenticados".

api.php — parte superior Sustituir 
// Bloque ACTUAL — eliminar en producción:
// $ip = $_SERVER['REMOTE_ADDR'] ?? '';
// if (!in_array($ip, ['127.0.0.1', '::1'], true)) { ... }

// NUEVO — añadir esto en su lugar:
session_start();
if (!isset($_SESSION['user_id'])) {
    http_response_code(401);
    echo json_encode(['error' => 'No autenticado. Inicia sesión primero.']);
    exit;
}
💡
Crear el primer usuario administrador

Ejecuta esto una sola vez desde la línea de comandos del servidor para crear el usuario inicial. Nunca uses la contraseña en texto plano en el código.

crear_admin.php — ejecutar una vez y luego eliminar Temporal 
<?php
$cfg  = require __DIR__ . '/assets/php/config.php';
$pdo  = new PDO("mysql:host={$cfg['host']};dbname={$cfg['name']}", $cfg['user'], $cfg['pass']);

$email  = 'admin@tuempresa.com';        // ← cambia esto
$pass   = 'una_contrasena_segura_2025!'; // ← cambia esto
$hash   = password_hash($pass, PASSWORD_BCRYPT);

$st = $pdo->prepare('INSERT INTO usuarios (nombre,email,password) VALUES (?,?,?)');
$st->execute(['Administrador', $email, $hash]);
echo "Usuario creado. Elimina este archivo ahora.";

Paso 3 — Configurar CORS correctamente Alto

El bloque de CORS actual de api.php permite el origen localhost. En producción debes sustituirlo por tu dominio real.

api.php y email.php — bloque CORS Sustituir 
// ACTUAL — permite localhost (solo válido para desarrollo):
// $origin = $_SERVER['HTTP_ORIGIN'] ?? '';
// if ($origin && (strpos($origin, '//localhost') !== false || ...)) {
//     header("Access-Control-Allow-Origin: $origin");
// }

// PRODUCCIÓN — lista blanca con tu dominio exacto:
$allowedOrigins = [
    'https://tudominio.com',
    'https://www.tudominio.com',
    // añade aquí subdominios adicionales si los necesitas
];
$origin = $_SERVER['HTTP_ORIGIN'] ?? '';
if (in_array($origin, $allowedOrigins, true)) {
    header("Access-Control-Allow-Origin: $origin");
    header("Vary: Origin");  // importante para caché HTTP
} else {
    http_response_code(403);
    echo json_encode(['error' => 'Origen no permitido']);
    exit;
}
⚠️
Aplica el mismo cambio en email.php

Ambos archivos (api.php y email.php) tienen el bloque CORS. Modifícalos los dos.

Paso 4 — Proteger o eliminar install.php Crítico

💣
install.php en producción = botón de borrar toda la base de datos.

Cualquier persona que acceda a https://tudominio.com/install.php puede reiniciar y borrar toda la base de datos con un clic. La opción más segura es eliminarlo.

Opción A — Eliminarlo (recomendado)

Terminal del servidor Eliminar 
rm /var/www/html/alquigest/install.php

Opción B — Protegerlo con contraseña HTTP (si necesitas conservarlo)

.htaccess — en la raíz del proyecto Añadir 
<Files "install.php">
    AuthType Basic
    AuthName "Zona restringida"
    AuthUserFile /ruta/segura/fuera/de/web/.htpasswd
    Require valid-user
</Files>

Para generar el fichero .htpasswd:

Terminal Shell 
htpasswd -c /ruta/segura/.htpasswd admin

Paso 5 — Credenciales de base de datos Alto

El archivo config.php usa root / root. En un servidor de producción el usuario root de MySQL tiene acceso total a todas las bases de datos del servidor.

Crear un usuario MySQL con permisos mínimos

MySQL — como root Ejecutar una vez 
-- Crear usuario específico para AlquiGest
CREATE USER 'alquigest_user'@'localhost'
    IDENTIFIED BY 'contraseña_muy_segura_aqui';

-- Dar solo los permisos necesarios (solo en su BD)
GRANT SELECT, INSERT, UPDATE, DELETE
    ON alquigest.*
    TO 'alquigest_user'@'localhost';

FLUSH PRIVILEGES;

Actualizar config.php

ℹ️
Nueva ubicación en v2.1+

El fichero de configuración se ha movido a assets/php/config.php. Todos los archivos PHP lo cargan automáticamente con la ruta correcta.

assets/php/config.php Editar 
return [
    'host'    => 'localhost',
    'port'    => 3306,
    'user'    => 'root',        // ← CAMBIAR
    'pass'    => 'root',        // ← CAMBIAR
    'user'    => 'alquigest_user',
    'pass'    => 'contraseña_muy_segura_aqui',
    'name'    => 'alquigest',
    'charset' => 'utf8mb4',
];
⚠️
No subas assets/php/config.php a un repositorio Git público.

Añade assets/php/config.php a tu .gitignore y usa variables de entorno del servidor o un archivo de configuración fuera del directorio web para guardar las credenciales.

Paso 6 — Eliminar la restricción de IP Medio

El bloque actual rechaza peticiones que no vengan de 127.0.0.1. En un servidor web los usuarios acceden desde IPs externas, así que hay que eliminar ese bloque y confiar en la autenticación de sesión (paso 2).

api.php, email.php y export.php Eliminar este bloque 
// ELIMINAR en producción:
$_ip = $_SERVER['REMOTE_ADDR'] ?? '';
if (!in_array($_ip, ['127.0.0.1', '::1'], true)) {
    http_response_code(403);
    echo json_encode(['error' => 'Acceso solo permitido desde localhost']);
    exit;
}

// La protección ahora viene de la sesión (ver Paso 2)
session_start();
if (!isset($_SESSION['user_id'])) {
    http_response_code(401);
    echo json_encode(['error' => 'No autenticado']);
    exit;
}

Paso 7 — Configuración de errores PHP Medio

PHP en modo desarrollo muestra errores en pantalla. En producción esto expone información interna al usuario. El bloque de errores en api.php ya oculta los mensajes de excepción; asegúrate también de que el php.ini del servidor esté configurado correctamente.

php.ini — ajustes de producción Verificar 
; Ocultar errores al usuario
display_errors = Off
display_startup_errors = Off

; Registrar errores en el log del servidor (para depurar)
log_errors = On
error_log = /var/log/php/alquigest_errors.log

; Nivel de reporte (mantener alto para que queden en el log)
error_reporting = E_ALL
✓
api.php ya está protegido

Los mensajes de error de base de datos ya usan error_log() en el servidor y devuelven solo "Error de base de datos" al cliente. Este comportamiento es correcto para producción.

Orden recomendado de implementación

1

Obtén el certificado SSL antes de tocar el código

Sin HTTPS el resto no sirve de nada. Usa Certbot en el servidor o el panel de control de tu hosting para activarlo. Verifica que https://tudominio.com funciona antes de continuar.

2

Crea el usuario MySQL y actualiza config.php

Hazlo antes de instalar la BD. Comprueba que la conexión funciona ejecutando install.php con las nuevas credenciales.

3

Implementa login.php y la tabla de usuarios

Crea la tabla de usuarios, ejecuta crear_admin.php para crear el primer administrador, y luego elimina crear_admin.php inmediatamente.

4

Actualiza api.php, email.php y export.php

Elimina el bloque de restricción de IP, añade session_start() y la comprobación de sesión, y actualiza el CORS con tu dominio real.

5

Actualiza AlquiGest.html para redirigir si no hay sesión

Al principio del script, comprueba si la API responde con 401. Si es así, redirige a login.html. La app ya gestiona el error de conexión; amplíalo para detectar la respuesta 401 específicamente.

6

Crea .htaccess y elimina install.php

Activa el redireccionamiento HTTPS y las cabeceras de seguridad. Elimina install.php definitivamente. Verifica con securityheaders.com que las cabeceras están correctas.

7

Haz una prueba de penetración básica

Intenta acceder a /api.php?action=getAll sin iniciar sesión: debe devolver 401. Intenta acceder a /install.php: debe devolver 403 o 404. Intenta hacer una petición desde otro dominio: debe ser bloqueada por CORS.

🎉
Una vez completados todos los pasos, AlquiGest estará listo para producción.

Los cambios en el código son relativamente pequeños. La mayor parte del trabajo es infraestructura (SSL, usuario BD, configuración del servidor). El código de la aplicación ya usa consultas preparadas, escapa HTML en las tablas y oculta errores internos.
